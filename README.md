# Pac-Man

## Description

A Python clone of the classic arcade game Pac-Man, built on top of pygame.
The maze is generated by an external `A-Maze-ing` package (assigned by another
group). The game features a main menu, persistent top-10 highscores, several
levels with a time limit, four ghosts with different chase behaviours, a
frightened mode after eating a super-pacgum, fruits, and a cheat mode that
the reviewer can use to exercise every feature quickly.

## Instructions

```
make install   # installs pygame, flake8, mypy
make run       # python3 pac-man.py config.json
make debug     # same, under pdb
make clean     # removes pycaches
make lint      # flake8 + mypy
```

You can also run it directly:

```
python3 pac-man.py config.json
```

The program takes exactly one argument: a JSON configuration file (any
filename is fine, the extension must be `.json`). Any error in the config
is logged and replaced by a safe default, the game does not crash.

### Controls

| Key                  | Action                                  |
|----------------------|-----------------------------------------|
| Arrows or WASD       | Move the player                         |
| Esc (in game)        | Pause                                   |
| Space (in pause)     | Resume                                  |
| M (in pause)         | Back to main menu                       |
| Space (in menu)      | Start a new game                        |
| H (in menu)          | Show top 10 highscores                  |
| I (in menu)          | Show instructions                       |
| Esc (in menu)        | Quit                                    |
| F1                   | Cheat: invincibility on/off             |
| F2                   | Cheat: skip current level               |
| F3                   | Cheat: freeze ghosts on/off             |
| F4                   | Cheat: +1 life                          |
| F5                   | Cheat: speed boost on/off               |

## Configuration

The config file uses JSON, plus lines starting with `#` are treated as
comments and stripped before parsing. Unknown keys are ignored and bad
values are replaced by their default. The accepted keys are:

| Key                       | Default              | Meaning                                  |
|---------------------------|----------------------|------------------------------------------|
| `highscore_filename`      | `"highscores.json"`  | Path to the highscores file              |
| `seed`                    | `42`                 | Seed used for the first level            |
| `lives`                   | `3`                  | Starting lives                           |
| `level_max_time`          | `90`                 | Seconds allowed per level                |
| `points_per_pacgum`       | `10`                 | Score for a normal pacgum                |
| `points_per_super_pacgum` | `50`                 | Score for a super-pacgum                 |
| `points_per_ghost`        | `200`                | Base score for the first ghost eaten     |
| `level`                   | (built-in)           | List of `{width, height}` for each level |

If the `level` array is shorter than the actual number of levels, the last
entry is reused for the rest. The game always runs for 10 levels.

## Highscore

Highscores are kept in a JSON file (path configurable via
`highscore_filename`). The file is loaded once when the game starts and
rewritten at the end of each run. Only the top 10 entries are kept, sorted
by score. We picked a flat JSON file because:

- It is portable (no SQLite/system dependency).
- It is human-readable, which is handy for debugging.
- The dataset is tiny (10 entries), so a fancy storage would be overkill.

Player names are clamped to 10 characters and only accept alphanumerics
and spaces. Scores must be non-negative integers (negative or non-integer
scores are dropped). If the file is missing, unreadable, or has invalid
JSON, the game starts with an empty list, no crash.

## Maze Generation

We do **not** generate the maze ourselves. We use the assigned
`A-Maze-ing` package located in `libs/mazegenerator/`. The integration
lives in `src/maze/adapter.py`:

- We instantiate `MazeGenerator` with `perfect=False` (Pac-Man compatible
  corridors).
- We translate its raw bitmask grid into our own grid format (`0`=empty,
  `1`=wall, `2`=pacgum, `3`=super-pacgum, `4`=pacman spawn).
- Super-pacgums are placed in the four corners, the spawn cell in the
  middle.
- Any exception raised by the generator is caught and converted into a
  clean error message, the game does not crash.

The first level uses the seed from the config (default `42`). Subsequent
levels use a fresh random seed each time, as required by the subject.

## Implementation

- `pac-man.py` - entry point, parses arguments, loads the config, starts
  the engine.
- `src/engine.py` - the `GameController` class. It owns the game state
  (entities, score, lives, level, mode) and runs the main loop with a
  dispatch on the current mode (menu, playing, paused, game over, ...).
- `src/maze/adapter.py` - wraps the A-Maze-ing package, exposes our grid.
- `src/models/entity.py` - abstract `Entity` base class + the
  `Direction` enum-ish class + the `ScorePopup` value object.
- `src/models/player.py` - `PacMan`, position is float for smooth
  movement, snaps to the grid when changing direction.
- `src/models/ghost.py` - `Ghost`, BFS pathfinding, four different
  targeting strategies (Blinky chases, Pinky ambushes 4 tiles ahead,
  Inky / yellow ambushes from behind, Clyde / orange wanders when close).
- `src/models/collectibles.py` - `Object` for pacgums, super-pacgums and
  fruits.
- `src/ui/game_view.py` - render the maze, the entities, the HUD and the
  score popups.
- `src/ui/menu.py` - the main menu screen (start / highscores /
  instructions / exit).
- `src/ui/screens.py` - pause, game over, victory, highscores and
  instructions screens, plus the `NameInput` helper.
- `src/utils/config_loader.py` - JSON-with-comments loader + validation.
- `src/utils/highscore.py` - load / save / sort the highscores.
- `src/utils/cheats.py` - `Cheats` class storing the cheat flags and
  mapping F1..F5 to actions.

## General Software Architecture

```
pac-man.py
  -- engine.GameController
        -- utils.config_loader          (load + validate config)
        -- utils.highscore.HighscoreManager
        -- utils.cheats.Cheats
        -- maze.MazeAdapter             (wraps libs/mazegenerator)
        -- models.PacMan
        -- models.Ghost  x4
        -- models.Object (pacgums, super-pacgums, fruits)
        -- ui.game_view.GameView        (HUD + render)
        -- ui.menu                      (main menu)
        -- ui.screens                   (pause / game over / ...)
```

The engine owns everything. Models are dumb (position + sprite + update +
move). The view is a render-only consumer of the model state. Menus and
screens are render functions plus an event handler that returns an
action. We chose a single state machine in the engine (the `mode` field)
rather than a stack of scenes, because the project is small and the
transitions are simple.
